# Architecture Design

This document explains how the Mindscape Journal system is designed and how it can scale to support large user bases while keeping LLM costs manageable and user data secure.

---

# System Architecture

The system follows a layered backend architecture.

```
Client
   ↓
API Routes
   ↓
Controllers
   ↓
Services
   ↓
Repositories
   ↓
Database
```

### Layer Responsibilities

Controllers
Handle HTTP requests and responses.

Services
Contain business logic such as emotion analysis and insights computation.

Repositories
Interact with the database using Drizzle ORM.

Database
Stores journal entries and analysis results.

---

# Data Model

Key tables:

### journal_entries

Stores user-written journal entries.

### journal_analysis

Stores results generated by the LLM.

```
journal_entries
      ↓
journal_analysis
```

Each journal entry can have a cached analysis result.

---

# How the System Scales to 100k Users

The system is designed to scale horizontally.

### Stateless API Servers

The backend API is stateless, allowing multiple instances to run behind a load balancer.

```
Load Balancer
     ↓
API Instance 1
API Instance 2
API Instance 3
```

This enables horizontal scaling as traffic increases.

---

### Database Scaling

For large workloads:

* Use PostgreSQL with connection pooling
* Add read replicas for analytics queries
* Use indexes for high-frequency queries

Example indexes already used:

* `journal_analysis.journal_id`
* `journal_analysis.emotion`

---

### Background Job Processing

Emotion analysis can be moved to a background worker queue (e.g., BullMQ or RabbitMQ).

```
User Request
      ↓
Store Journal
      ↓
Queue Analysis Job
      ↓
Worker calls LLM
```

This prevents LLM latency from blocking user requests.

---

# How LLM Cost is Reduced

LLM calls are one of the most expensive operations.

To reduce cost:

### 1 Store Analysis Results

The system stores analysis results in the `journal_analysis` table.

If a journal has already been analyzed, the cached result is returned instead of calling the LLM again.

```
Check DB
   ↓
If analysis exists → return cached result
If not → call LLM
```

---

### 2 Use Smaller Models

Emotion classification does not require very large models.

Efficient models such as:

* `llama-3.3-70b-versatile`
* `gpt-oss-120b`

provide strong performance with lower cost.

---

### 3 Batch Processing

For large datasets, journal entries can be analyzed in batches rather than individually.

---

# Caching Repeated Analysis

Repeated analysis is prevented using a caching strategy.

### Database Cache

The `journal_analysis` table acts as a cache.

```
journalId → analysis result
```

Before calling the LLM:

```
check existing analysis
```

If found, return cached data.

---

### Optional Redis Cache

For high-traffic systems, Redis can cache recent analysis results.

```
Redis
   ↓
DB fallback
```

This reduces database load and improves response time.

---

# Protecting Sensitive Journal Data

Journal entries may contain highly personal information, so protecting user privacy is critical.

### 1 Authentication

JWT authentication ensures only authorized users can access their journal entries.

---

### 2 Access Control

Users can only access their own journal entries.

Authorization checks verify:

```
req.user.id === journal.userId
```

---

### 3 Encryption

Sensitive data can be encrypted at rest.

Options include:

* PostgreSQL column encryption
* Application-level encryption for journal text

---

### 4 Secure Transport

All communication between clients and the API uses HTTPS.

---

### 5 Data Minimization

Only required fields are stored in the analysis table.

The system does not store full LLM prompts or unnecessary metadata.

---

# Future Improvements

To further improve scalability and reliability:

* Redis caching
* Background job queues
* Rate limiting for LLM endpoints
* Streaming LLM responses
* Event-driven architecture
* Monitoring and logging

---

# Conclusion

The system architecture prioritizes scalability, cost efficiency, and user privacy. By caching LLM results, using efficient models, and designing a modular backend architecture, the platform can support large user bases while maintaining good performance and manageable infrastructure costs.
